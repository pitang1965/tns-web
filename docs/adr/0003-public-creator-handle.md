# ADR-0003: 公開旅程の作成者を匿名ハンドルで表示する

## ステータス

承認済み

## コンテキスト

公開旅程の一覧では、作成者を「作成者: ◯◯」として表示している。表示名は Auth0 セッションの `owner.name` / `owner.email` を元にしており、ログイン方法によって見え方が割れていた。

- メール/パスワードログイン：`owner.name` がメールアドレスになりがちで、`maskEmail` によって `ta****@***` のようにマスクされる
- Googleログイン：`owner.name` が本名の表示名（例：山田太郎）になり、**マスクされずそのまま表示**される

調査の結果、より深刻な問題が判明した。`getPublicItineraries()` は `owner` オブジェクト（`id` / `name` / `email`）を**そのままクライアントへ送信**しており、公開ページのデータ payload に**生のメールアドレス・本名が含まれていた**。表示時のマスクは見た目だけで、DevTools やソースから生PIIを読み取れる状態だった。

作成者表示の目的を整理したところ、「**誰が作ったかという帰属・親しみを示す**」ことであり、本名やメールアドレスの正確な値を伝える必要はない、と確認した（信頼・帰属の表示が目的）。本サービスにユーザー間コミュニケーション機能は無く、自分の旅程は「自分の旅程」タブで別途確認できる。

検討した案：

- **案A**: 表示ロジックだけ修正（本名もマスクする）。最小変更だが、生PIIは payload に残り続けるため privacy 目的を達成しない
- **案B**: 実名・メールの断片を残した半匿名表示（例：`山🐓さん` / `ta🐼さん`）。ログイン方法ごとに頭の見え方が割れる非一貫性を再生産し、実名断片も残る
- **案C**: 完全匿名の決定的ハンドル。`owner.id` から決定的に「接頭辞＋動物＋絵文字」を割り当て、サーバー側で生成して payload から生PIIを除去する

## 決定

**案C** を採用する。公開旅程一覧（`getPublicItineraries` → `PublicItineraryItem`）について：

1. **作成者ハンドルをサーバー側で生成**する。`sha256(owner.id)` のバイト列から接頭辞・動物のインデックスを算出し、決定的に割り当てる。同一の作成者には常に同じハンドルが付く。
2. **公開リストの payload から `owner.name` / `owner.email` / `owner.id` を除去**し、クライアントには生成済みのハンドル文字列のみを渡す。
3. 表示フォーマットは `作成: 旅人コッコ🐓さん`（接頭辞＋動物名＋絵文字に密着、末尾に「さん」）。動物ハンドルであり人物名ではないため、ラベルは「作成者」ではなく「作成」とする。
4. 名前空間は接頭辞 約40語 × 動物 約30種 ≒ **約1,200通り**。
5. **衝突は許容する**。別人が同じハンドルになり得るが、コミュニケーション機能が無く一覧カードは「その旅程の作者」を示すだけなので実害は小さい。一意性の保証（登録テーブル）やサフィックス付与は行わない。
6. `owner` や `owner.id` が無い旧データのみ `名無しさん` にフォールバックする。

## 理由

- 報告された「ログイン方法で作成者表示が割れる」非一貫性を、形式を一定化して根本から解消する
- サーバー側で生PIIを除去するため、payload からメール・本名が漏れる privacy ホールを実際に塞ぐ（マスクのような見た目だけの対策ではない）
- ハンドル生成材料（name/email/id）をクライアントに残さないため、ハンドルは改ざん・逆引き不能
- `owner.id` からの決定的生成なので、スキーマ変更・データ移行・状態管理が不要。実装は表示経路とサーバー projection に限定できる
- 衝突許容により、ステートレスのまま可愛らしい見た目（記号サフィックス無し）を保てる。本サービスの規模では実際の衝突は当面ほぼ起きない

## トレードオフ

- 別人が同じハンドルになる衝突が原理的に起こり得る。一意性は保証しない（帰属の弱さは許容範囲と判断）
- 動物ハンドルは「誰が作ったか」の温度感を強くは伝えない。正確な身元を見せる用途には使えない（それが目的ではない）

## 適用範囲と残課題

- 本ADRの対象は**公開旅程一覧（`PublicItineraryItem`）のみ**（スコープS1）。
- **詳細ページの payload リーク（スコープS2）は対応済み**。当初は `useItineraryAccess` が `owner.id` / `sharedWith[].id` をクライアントで使って所有者・共有判定をしていたため、`owner`（name/email を含む）と `sharedWith` が詳細ページの payload に載っていた。
  - **解決方法**：所有者・共有判定をサーバー側で算出し、`isOwner` / `isSharedWith` の boolean のみをクライアントへ渡す形へ作り替えた。`owner` と `sharedWith` は詳細ページの payload から完全に除去した。
  - **対象経路**：`GET /api/itineraries/[id]`（`toDetailItinerary()` で生成）と `GET /api/itineraries/[id]/day/[dayIndex]`。両者は元々サーバー側で `canAccessItinerary` による認可ガード（不許可は404）を行っていたため、判定材料の生PIIをクライアントへ送る必要がなかった。
  - **型**：`DetailItineraryDocument`（`owner` / `sharedWith` を除き `isOwner` / `isSharedWith` を持つ）を新設。`useGetItinerary` / `useGetItineraryDay` / `useItineraryAccess` / `ItineraryDetail` がこれを消費する。
  - **キャッシュ purge**：IndexedDB キャッシュ（`itineraryCache.ts`）の `DB_VERSION` を 2 に上げ、upgrade 時に旧ストアを作り直すことで、既存ユーザーのブラウザに残る生PII入りキャッシュを破棄する。
  - **stale cache 対策**：キャッシュされた `isOwner` はキャッシュ時点のセッションに基づくため、`useItineraryAccess` で「現在ログイン中である場合のみ true」とガードし、ログアウト後に所有者UIが誤表示されないようにした（実際の変更操作はサーバー側で owner.id により別途ガードされる）。

## 注意

公開リストの payload から `owner.name` / `owner.email` / `owner.id` を落としているのは「privacy のための安全装置」である。安易に owner をそのまま渡す実装に戻すと、生PIIが公開ページに再び露出する。変更時はこのADRを参照すること。
