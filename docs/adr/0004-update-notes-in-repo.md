# ADR-0004: 更新情報をリポ内データファイルで管理し、デプロイ連動・管理UIなしとする

## ステータス

承認済み

## コンテキスト

一般ユーザーに向けて、アプリに加えられた変更（新機能・不具合修正・その他変更）を平易な日本語で伝える「[更新情報](../../CONTEXT.md)」を新設する。あわせて、変更があったことを未読バッジで知らせる。

このサービスの既存コンテンツ（旅程・車中泊スポット・スポット投稿）は、いずれも **MongoDB + 管理UI** で管理されている。管理者は `/admin/*` のページからデプロイなしで登録・編集・承認ができる。そのため「更新情報も同じく DB + 管理UI で作るべきか」が自然な論点になった。

一方で更新情報には他コンテンツと異なる性質がある：

- **changelog はリリース（デプロイ）と1対1で対応する。** 「何が変わったか」は、変わったコードが本番に出た時にしか確定しない。
- **掲載対象は一般ユーザー向けに取捨選択・書き換えた文章であり、ユーザー数や状態を持たない静的な読み物。** 集計・検索・権限分岐などの DB の利点をほとんど必要としない。
- **執筆者は実質的に開発者（兼管理者）ひとり。** 変更を加える PR の中で、その変更の説明も一緒に書ける。

検討した案：

- **案A**: MongoDB + 管理UI（model + API + `/admin` ページ）。他コンテンツと一貫し、デプロイなしで投稿・修正できる。一方、新規モデル・API・管理画面の実装と保守が必要で、更新情報という「変更そのもの」が、それを生んだコード変更から切り離される。
- **案B**: リポ内データファイル（`src/data/updateNotes.ts`）。型付き配列として保持し、変更を加える PR の中で項目も記述する。git で版管理され、DB/API/管理UI は不要。追加・修正には再デプロイが必要。
- **案C**: Markdown コンテンツ + レンダラ依存（`react-markdown` 等）。執筆は最も楽だが、プロジェクトに存在しない描画依存を追加し、カテゴリアイコンやグルーピングの型安全性が下がる。

## 決定

**案B** を採用する。

1. 更新情報は **リポ内データファイル** `src/data/updateNotes.ts` に、型付きの配列として保持する。
2. データ型は日付グループ単位とする：`UpdateEntry { date: 'YYYY-MM-DD'; items: UpdateItem[] }`、`UpdateItem { category: 'new'|'fix'|'other'; text: string; href?: string; linkLabel?: string }`。`YYYY-MM-DD` は文字列比較がそのまま日付比較になるため、これを比較キーにも用いる。
3. **changelog のみを扱い**、メンテナンス予定・障害・キャンペーン等の告知は対象外とする（将来必要なら別概念「お知らせ」として分離）。
4. 各項目は **1リンクまで**（末尾リンク）とし、Markdown/HTML パーサ等の描画依存は追加しない。
5. 表示は専用ページ `/updates`（全件・公開）と、ログイン時ダッシュボードの最新数件プレビューで行う。
6. 既読管理は **localStorage** に最終既読日付を保持し、`最新entry日付 > 既読日付` で未読バッジを点灯する（端末ごと、バックエンド不要）。

## 理由

- 更新情報はリリースと1対1で対応するため、**変更を加える PR の中で項目も書く**運用が自然で、changelog と実装の乖離を防げる。
- 静的な読み物であり、DB が提供する集計・検索・権限分岐をほとんど必要としない。案A の model + API + 管理画面は、得られる価値に対して実装・保守コストが過大。
- 描画依存を増やさないことで、バンドルと保守対象を最小に保つ。カテゴリ（アイコン・色）とグルーピングは型付き TS で安全に扱える。
- 既読状態は changelog バッジ用途であり端末をまたぐ厳密さは不要。既存の `InstallBanner` と同じ localStorage パターンに揃う。

## トレードオフ

- 更新情報の追加・誤字修正にも **再デプロイが必要**。デプロイなしでの即時投稿はできない（changelog の性質上、許容できると判断）。
- 他コンテンツ（DB + 管理UI）と**管理方法が一貫しない**。「これだけ管理画面で編集できない」理由を本ADRで明示することで、将来の混乱を防ぐ。
- 既読管理が端末ごとのため、別端末では既読が引き継がれない。また日付（`YYYY-MM-DD`）粒度のため、**同一日に2回更新した場合の2回目はバッジを再点灯しない**（日次粒度の changelog として許容）。

## 注意

将来、更新情報を「お知らせ」（メンテ・障害・キャンペーン告知）へ拡張したくなった場合、それらは非changelog（スケジュール日時や掲載期間を持つ）であり本ADRの前提と異なる。その時点で改めて DB + 管理UI（案A）への移行を ADR として再検討すること。本ADRの決定は「changelog のみ」という範囲に限った判断である。
