# ADR-0006: モバイルアプリ向けにバージョン付き専用API(v1)を設ける

## ステータス

承認済み

## コンテキスト

新規ユーザー獲得を目的としたモバイルアプリ(Expo/React Native、別リポジトリ)を計画している。アプリの中核機能は「近傍スポット検索」(現在地から近い車中泊スポットの地図・一覧表示)で、詳細閲覧はブラウザでWebの詳細ページ(`/shachu-haku/[spotId]`)を開く薄い設計とする。

アプリがサーバーのデータを必要とするが、既存APIには問題がある。

- ストアに配布したアプリは強制アップデートできず、古いバージョンが世に残り続ける。Webと違い「明日APIを変えても誰も困らない」が成り立たない
- 既存の内部API(`/api/camping-spots` 等)はWebフロントの都合で自由に変更されており、アプリが直接依存するとWeb都合の変更が旧アプリを壊す
- そもそも「現在地から近い順にスポットを返す」エンドポイントは存在しない(`/api/camping-spots` は新着順最大100件、`/api/camping-spots/nearby` はスポット周辺のトイレ・コンビニ・入浴施設を返す別物)

## 決定

`/api/v1/` 配下にモバイルアプリ専用のエンドポイント群を新設する。**v1は後方互換を壊さない**ことを契約とする。Web用の内部APIは従来通り自由に変更してよい。

最初のエンドポイントは**全スポットの軽量一覧 `/api/v1/spots`** とする(2026-07時点で約1,900件、最小フィールドでgzip後~70KB)。近傍判定・距離計算・ソートはアプリ側で行う。位置引数を持たない単一URLのため、`Cache-Control: s-maxage` によりVercel CDNでほぼ全リクエストを吸収でき、サーバー・DB負荷対策(レート制限の代替)を兼ねる。副次効果として、アプリが全件をキャッシュすれば圏外でも全国のスポットを参照できる。

件数増大でペイロードが許容を超えた場合は、近傍検索エンドポイント(`$near`、2dsphereインデックスはモデル定義済み)を**v1への追加**として導入する(既存契約は壊さない)。

## 理由

- アプリの生存期間とWebのリリースサイクルを分離できる。旧バージョンのアプリが動き続ける
- 「アプリに何を見せるか」(名前・種別・距離・料金のみ)という誘導設計上の境界線を、APIレスポンスの形として強制できる
- 認証なし公開APIであるため、レスポンスを最小限に絞ることはデータのただ乗り対策にもなる

## トレードオフ

- Web内部APIと部分的に重複したエンドポイントを持つことになり、スポットのスキーマ変更時は2系統への影響を考慮する必要がある
- 「v1を壊さない」制約は将来の設計変更の自由度を下げる。破壊的変更が必要になったら v2 を並設し、v1 は旧アプリの利用が十分減るまで残す

## 注意

このAPIは配布済みモバイルアプリが依存する。`/api/v1/` 配下のレスポンス形式を変更・削除する前に、必ずこのADRとアプリ側の互換性を確認すること。

## 追記(2026-07): 固定APIキーによる保護

`/api/v1/*` は `src/proxy.ts`(Next.js 16のproxy、旧middleware。Auth0処理と同居)で固定APIキー(`x-api-key` ヘッダー)により保護する。

- サーバー側環境変数 `SPOTS_API_KEY` とヘッダー値が一致しなければ 401。**未設定でも 401**(fail-closed。アプリ未リリース時点で導入したため段階導入は不要と判断し、設定漏れで保護が静かに無効化されない側に倒した)
- モバイル側は `EXPO_PUBLIC_SPOTS_API_KEY` に同じ値を持ち、バンドルに埋め込まれる
- ルート内でなくプロキシで検証する理由: 本APIは `s-maxage` でVercel CDNにキャッシュされ、キャッシュHIT時はルート関数が実行されない。プロキシはキャッシュ前段で毎回走るため、キャッシュ済みレスポンスもキー無しには返さない
- これは本物の認証ではなく匿名・野良アクセスを弾く「段差」。キーはアプリバイナリから抽出可能で、全クライアント共通キーのため利用者単位のBANもできない
- キーをローテーションすると旧バージョンアプリが一斉に401になる。ローテーション時はプロキシを新旧2キー許容に拡張して移行期間を設けること
